Network AI-Assistant · v4 (Persona-aware)

Network AI-Assistant · 在 LLM 環境裡用的 EnGenius Cloud 工具集

這是在 LLM coding 環境（Claude Code）中使用的 EnGenius Cloud 管理工具集，不是雲端管理平台 GUI 內嵌的 AI chatbox。 Claude Code 在對話中編排 RD 的 13 個 data skill + dashboard-builder skill + persona，依使用者問題自動選輸出形式：文字 / dashboard / 動手操作。

本頁聚焦 Capability 1（Dashboard Builder）。整體 3 個 capability 全貌見下方 §00 產品全貌。「即時組成」是我們的 wedge——不是預先做好的模板，而是訪客問問題的當下，AI 才決定 dashboard 長什麼樣。

00產品全貌 · Network AI-Assistant 是什麼

一個容易被誤會的點：這套東西不是在 EnGenius Cloud 網頁裡加個聊天視窗。 它是在使用者自己的 Claude Code（或其他 LLM coding 環境）裡跑的 EnGenius Cloud 管理工具集。 AI 在使用者本地的終端機環境裡，透過 skill 呼叫 EnGenius Cloud API，產出文字回答、dashboard 或執行動作。

這個 ≠ 那個 ✅ 是：你在 Claude Code 終端機裡用自然語言對話，AI 透過 skill 呼叫 EnGenius Cloud API，回你文字 / 生成 dashboard HTML / 動手操作雲端設定
❌ 不是：在 cloud.engenius.com 網頁裡開一個聊天視窗（那是 Line 1 路線——「Cloud-Embedded AI」，目前未推進）

3 個 Capability · Dashboard Builder 是其中之一

整套 Network AI-Assistant 由 3 個 capability 組成。本頁聚焦 Capability 1，其他 2 個跨 page 或跨文件說明。

Capability 做什麼 狀態
1 · Dashboard Builder ⭐ 本頁主題 對話即時組裝視覺化 dashboard（10 widget · 5 validated 情境 · 3 語言 · light/dark theme） ✅ 已可 demo
2 · Conversational Admin persona-driven AI 用「網管顧問」語氣回應、翻譯術語、主動提醒、自動判斷該回文字 / dashboard / 動手 （§02c） 🔜 等 RD 改 SKILL.md
3 · Action Operator 現場動手 觸發實體動作：閃 LED 定位設備、踢可疑 client、cable diagnostics、遠端 reboot 🔴 卡 RD P0 ops （rd-priorities）

誰用 / 在哪用

主要使用者：

  • SI / Partner / 顧問 — 代 SMB 客戶問問題、生報告、做網路審查
  • 技術背景的 SMB IT — 自己會用 AI coding 工具的客戶（少數，但增長中）
  • EnGenius 內部 booth team — 展會 demo 用
  • 未來：EnGenius 內部 support / FAE — 加速處理客戶 ticket
SMB 終端客戶（餐廳老闆 / 診所經理 / 學校主任）不直接打字——他們透過 SI / 自家技術 IT / EnGenius 支援接觸到結果。但 persona §1 的 SMB-friendly voice 確保不管誰用，輸出語氣對 SMB 客戶友善。

基本使用流程

  1. SI / partner 在自己電腦開 Claude Code（CLI / Desktop / VS Code 擴充）
  2. 確保 api-skills/（RD 給的 EnGenius Cloud skill 套件）+ dashboard-builder/skill/ 已安裝，環境變數 MANAGE_SYSTEM_URL + API_KEY 設好
  3. 跟 Claude 對話：「我們客戶今天網路怎樣？」「幫我看 X 客戶 7F 的 wifi 為什麼慢」「列一張到期 license 給我看」
  4. Claude 依 persona 升級條件 決定輸出形式：
    • 單一答案 → 文字回應（直接 paste 回給客戶）
    • 多維度 / 跨 org / 趨勢 → 生 dashboard HTML（分享連結或截圖）
    • 明確指令 → 動手操作（先確認再動）
  5. 產出傳遞給客戶 — 文字直接複製、dashboard HTML 分享連結或截圖、操作後回報結果

為什麼選「LLM coding 環境」而不是 cloud GUI 嵌入

  • 更新成本低 — 改 persona / 加新 skill 不用動 cloud frontend 發版
  • 能力更大 — Claude Code 可呼叫 shell / 多 skill 組合 / 寫檔，cloud GUI chatbox 只能讀
  • 定位差異化 — 跟其他廠商的「cloud GUI + ChatGPT 翻譯」明顯不同，wedge 更深
  • SI / partner 友善 — 他們本來就在用 Claude Code 處理多家客戶，這個工具直接進工作流

未來 Line 1（cloud-embedded chatbox）路線可以另外推進，但這次 focus 是 Line 2（LLM 環境工具集）。兩條路不衝突，但目前不混。

01兩個無法妥協的原則

原則 1 資料一定真實——所有 dashboard 數字必須來自 staging API，不可硬編。
違反 → 觀眾識破是 mockup。
原則 2 組裝一定即時——dashboard 結構在對話中才決定，不可全套打包成 skill 預設出來。
違反 → 變成普通 SaaS 預設 view，wedge 故事死。

3 層之間的關係

Playbook ➡ Orchestration Claude 啟動對話時讀 playbook（widget docs / spec examples / scenario candidates），知道「能用什麼 / 過去怎麼用」。改 markdown 就能微調 Claude 行為。
Orchestration ➡ Primitives Claude 用 shell 呼叫 RD data skill 拿 JSON、寫 spec、呼叫 compose 出 HTML。Primitives 不知道是誰呼叫它的。
Primitives ⇆ Primitives 各 primitive 互不相識——data skill 不知道 widget 存在、widget 不知道 cloud API 存在。Theme token 是唯一跨 widget 的耦合（且是 read-only）。

為什麼分這 3 層 vs 不分

有些 SaaS 會把所有業務情境包成「scenario skill」（例如 org-health-overview = 撈 5 個 API + 預設 view）。我們故意不做這層，因為：

  1. 觀眾識破——「預設好的 view」跟現有 cloud GUI 沒兩樣，wedge 故事死。
  2. 加新情境成本高——每加一個情境就要寫一個 scenario skill + 部署。
  3. Claude 已經很會了——對話 + playbook docs 足以讓它即時組合，不需要寫死。

反過來說，如果情境需要「確定性」（同樣的問題每次出來完全相同的 dashboard），就退化為「直接 compose 既有 spec」——但這應該是例外，不是常態。

02cAI 網管人格層 · Persona

Persona = Claude 的「voice + 判斷規範」。讓所有 skill 的回應統一語氣、自動選對輸出形式。 這是 wedge 故事從「AI 動態生 dashboard」升級到「你的專屬 AI 網管，會自己判斷要文字回 / 開 dashboard / 動手操作」的關鍵層。

為什麼這層必要 沒有 persona，Claude 回客戶話會是「3 台 AP offline」——SMB 客戶（餐廳老闆 / 診所經理）聽不懂。
有了 persona，同一份 API 結果變成「廚房那台 AP 從 14:23 開始失聯，可能是斷電。要我幫你查嗎？」——直接拉開 wedge 跟其他「LLM + API」工具的差距。

兩層知識架構

Persona 不是孤立的一份文件——它是「給 Claude 的知識體系」的第 1 層。目前定義 2 層：

① persona.md 已 commit · 356 行
Voice + 輸出格式 + 升級條件。Claude 每次被 skill 觸發都載入。內含 4 條 voice 原則、15 個詞彙翻譯、6 個 SMB vertical 適應、6×3 升級矩陣、3 種輸出模板。
② house-rules.md 未來 P2 · ~120 行
EnGenius 品牌觀點。HVS 分數解讀區間、永遠不推薦的反 pattern、平台特有 gotcha（例：cloud show offline 但現場正常 = cloud-agent 沒同步）。需 RD + 業務 + PMM 共寫。
為什麼不把 troubleshooting 寫死成「網管百科」？ Claude 訓練資料已含一般網管知識；EnGenius 產品細節在 Help Center / Gitbook 才是 single source of truth，但那是另一個系統，目前不在本專案 scope。若未來要讓 Claude 能查 EnGenius 內部文件，需要單獨打通 RAG 整合（屬於 future work）。重點是：硬抄到 skill ref → 兩份 doc drift；token 成本高；過度約束會殺掉「即時判斷」的 wedge。

升級條件矩陣 · text / dashboard / action

Persona 最核心的「智慧」就是這張表。Claude 每次回應前掃過 6 條訊號，選最符合的那一欄輸出。「AI 自己判斷該開 dashboard」比「AI 生 dashboard」更厲害一個 level——這是 wedge 故事的升級點。

觸發信號 → 文字回應 → 開 dashboard → 動手操作
設備 / 項目數量 1-5 個 6+ 個或跨 org —
時間維度 「現在」「剛剛」 「最近一週」「趨勢」「比較」 —
問題形式 單一答案 / yes-no / 解釋為什麼 多維度交叉、分布、排名 明確指令
觀眾意圖 私下問、確認、學習 要分享、要帶到會議、要決策 「幫我關掉 / 踢掉 / 重開」
資料新鮮度 不在乎、cached 即可 想 live 看到變化 —
使用者明示 「我只要一句話」 「拉一張圖」「show me」 動詞祈使句

Tie-break：超過 1 條訊號落在同一欄 → 走那欄。落在兩欄之間 → 走文字，並結尾問「要拉 dashboard 嗎？」

為什麼 persona 不破壞 wedge 故事

原本對外故事是「AI 動態生 dashboard」。加上 persona 後，故事升級成：

升級後的 wedge 「你的專屬 AI 網管 — 他會回答你、給建議、必要時生 dashboard 給你看。」
Dashboard 從主角變成工具。AI 網管才是主角——他自己判斷什麼時候用什麼形式回應。
這比「AI 是 dashboard generator」更有人味，也更難被其他「LLM + API」工具複製。

03什麼是「Widget」？

每塊橘色虛線分區 = 一個 widget。一張 dashboard 由多個 widget 疊出來。

6 個 widget（目前）

alert 警示橫條。severity 分 critical / warning / info / success。body 可寫死或用 compute function 從 JSON 算出來。
kpi_grid KPI 卡片列。支援 2-6 張卡，每張可點擊跳到另一個 widget，數字可從 JSON 路徑直接讀或用 compute function 算。
table 表格。內建 filter chips、可展開 row 看 detail、可排序、可接收跨 widget 事件（topology 點 device → table 自動展開那一行）。
bar_list 橫條圖。多組 bar，可以分群（type 一群、license 一群）。
chip_strip 標籤條。用來秀「empty networks」「stale services」等不需要表格化的小量項目。
topology_tree 樹狀圖。Org → Network → Device 階層、可展開、被 RBAC 擋的 org 會顯示 403/402 badge。

換情境 = 換 spec JSON，不用重寫 dashboard

同樣這 6 個 widget，可以組出完全不同的 dashboard：

// 情境 A：Org 健康總覽（現有版）
{
  "title": "Main_Org · 健康總覽",
  "sections": [
    { "widget": "alert", "severity": "critical", "title": "License 全面過期警示" },
    { "widget": "kpi_grid", "items": [/* 4 KPIs */] },
    { "widget": "table", "title": "Device Fleet", ... },
    { "widget": "topology_tree", "title": "Network Topology" }
  ]
}

// 情境 B：離職員工權限大掃除（新情境）
{
  "title": "Off-boarding Compliance Audit",
  "sections": [
    { "widget": "alert", "severity": "warning", "title": "5 人超過 1 年沒登入仍持有 network 權限" },
    { "widget": "kpi_grid", "items": [/* stale users / risk accesses */] },
    { "widget": "table", "title": "Stale Members（only）", ... }
  ]
}

不用碰任何 widget 程式碼，只要寫新 spec JSON → compose.py → 出新 dashboard。

03bLibrary 哲學：按需擴張，不一次蓋好

Widget library 不是一次盤點所有可能的視覺化、提前 build。每次有新情境，才從中萃取出真正用得到的 widget。這樣保證每個 widget 都有 production-grade 的真實使用案例，避免「為了完整性建出沒人用的元件」。

判斷「該不該加 widget」的 3 個問題

  1. 現有 widget 真的做不到嗎？很多時候 bar_list / table + compute_fn 就能解，不用新 widget。
  2. 會被重用嗎？如果只有這次 demo 用到，建議用一次性 HTML 嵌入而非變 widget。
  3. 有沒有設計 token 還沒覆蓋的視覺？如果有，先補 token、再寫 widget，避免硬碼顏色。

Library 不是越大越好——8 個經過真實使用、彼此互不重疊的 widget，比 30 個「以防萬一」的 widget 對 Claude 更友善：選擇少 → 不會選錯 → 視覺更一致。

04怎麼用 Dashboard Builder Skill

三個情境，每個情境都是 Claude Code 對話中觸發。Skill 本身就是一個 Python 入口加一堆 HTML 零件。

用法一：Claude 在對話中自動跑（典型 demo 場景）

  1. 訪客問問題
    「幫我看 Main_Org 健康狀況」
  2. Claude 跑 data skills、把結果寫進 live-data/
    並行呼叫 get_user_orgs / get_hierarchy_views / get_inventory / get_licenses / get_org_memberships_overall，輸出到 prototype/live-data/*.json
  3. Claude 寫一份 spec JSON
    大概 100-150 行的 JSON，描述「想要幾個 widget、按什麼順序、用哪份 live-data」。範例：dashboard-builder-skill/examples/org-health.spec.json
  4. Claude 呼叫 compose.py
    python prototype/dashboard-builder-skill/scripts/compose.py \
  --spec my-spec.json \
  --out dashboard-builder/canvas-name.html
    產出單一 HTML 檔（46KB），自包含所有 widget CSS + JS + runtime。
  5. 瀏覽器自動打開 preview
    Dashboard 載入後立即 fetch live-data/、render 所有 widget，每 5 秒 poll 一次更新。
  6. Claude append manifest log entry
    generated-log.html 立即跳出新 entry，操作員可從那邊回追今天 demo 過什麼。

用法二：手動 compose 已有 spec（重現舊 demo）

# 重新跑 examples/org-health.spec.json 產生新 timestamp 的 canvas
python prototype/dashboard-builder-skill/scripts/compose.py \
  --spec prototype/dashboard-builder-skill/examples/org-health.spec.json \
  --out dashboard-builder/org-health-test.html

# 跑完打開 preview server
open http://localhost:58778/dashboard-builder/org-health.html

用法三：刷新 live-data 但保留 canvas（最常用）

展會中要 demo「live 數字會跳」效果：

# 1. 操作員執行 refresh 腳本（重撈 staging API）
bash dashboard-builder/scripts/refresh-all.sh

# 2. Dashboard 已開著的話，5 秒內自動 reflect 新數字
# 不用重 compose、不用重 reload

用法四：切視覺主題 + 切語言（i18n）

每個 spec 都有 locales 區塊（zh-TW / en / ja）。用 --locale flag 切換：

# 預設（spec 的 zh-TW）
python dashboard-builder/skill/scripts/compose.py \
  --spec dashboard-builder/skill/examples/org-health.spec.json \
  --out dashboard-builder/test.html

# 英文版
... --locale en

# 日文版（日本展會用）
... --locale ja

# 日文 + dark mode 雙堆疊
... --locale ja --theme dark

→ Gallery 上每張 dashboard card 下方都有「view in 中 / EN / 日」切換鍵，點任一個直接看對應語言版本。日文版作為日本 booth 主秀，需要 native speaker review 過後正式上場。

04bDemo Readiness · validated path 紀律

Live demo 沒有「我來試試看」這種選項——每個要在 booth 演出的情境都必須事先成為 validated path。沒驗證的不准上場。這是「wedge 故事即時組」的相反面：外觀即時、底層保底。

Validated Path 的 3 個必要條件

一個情境要被當作 validated path，必須三件事都完成：

✓
1. 有對應的 spec.json 存在於 skill/examples/

不能用 Claude「現場從零寫 spec」當主秀。Claude 可以微調既有 spec（改 title、改 KPI 標籤、加 1-2 個 widget），但骨幹必須是已存在的 JSON 檔。目前 5 個 validated：S2 / S3 / S4 / S5 / S7。

✓
2. 已用 compose.py 成功跑過、產出可看的 HTML

產出的 HTML 必須能在瀏覽器開、視覺正常、所有 widget 渲染正確、互動 work、auto-poll work。任何一項壞，這個情境不准上 booth。

✓
3. 連到的每個 op 都在 RD 已 supported 範圍

spec footer 的 ops_used 列出來的每個 op 必須能跑（在 rd-priorities.md 的「已 supported」清單裡）。任何「規劃中」🔴 的 op 都不算。

兩個關鍵 script · 每個 validated path 都會跑

整套 demo 流程靠這兩個 script 撐起來——一個撈雲端、一個產 HTML。

refresh-all.sh ~14 秒

做什麼：呼叫 6 個 RD data skill，把雲端最新狀態存進 live-data/*.json。

什麼時候跑：展會開始前 5 分鐘；或展中想 demo「資料剛剛變了」時。

# 展前準備
bash dashboard-builder/scripts/refresh-all.sh

# Output:
#   live-data/orgs.json         (5 orgs)
#   live-data/hierarchy.json    (10 networks)
#   live-data/inventory.json    (10 devices)
#   live-data/licenses.json
#   live-data/memberships.json  (11 members, 5 stale)
#   live-data/topology.json     (cross-org)
compose.py ~200 ms

做什麼：吃一份 spec.json + 讀 widgets/theme/runtime，組成一份自包含 HTML。

什麼時候跑：每次驗證情境時；展中如果要重 compose 用客製化 spec。

# 從既有 spec 生 dashboard
python dashboard-builder/skill/scripts/compose.py \
  --spec dashboard-builder/skill/examples/org-health.spec.json \
  --out /tmp/test.html

# 換 dark mode
... --theme dark

新情境的 Validation Pipeline · 6 步

有了新情境想法 → 走完這 6 步才能上 booth：

  1. 確認需要的 op 都能跑
    查 rd-priorities.md，把這個情境會用到的 op 都列一遍。如果有任何一個在 P0 待完成或 troubleshoot skill 內 → 暫停，等 RD 補完才能繼續。
  2. 寫 spec.json（複製近的範例改）
    不要從零寫——挑 examples/ 裡最近的（角度、widget 組合接近的），複製、改 title / KPI compute_fns / data source / widget 選擇。
  3. 跑 compose.py 產 HTML
    先用 --out /tmp/test.html 試水溫。視覺看起來對才搬到 dashboard-builder/<name>.html。
  4. 驗證互動 + auto-poll + cross-widget event
    點 KPI 卡跳轉、展開 row、按 Refresh、5 秒 poll 自動更新、topology tree 點 device 跳 table——每一項都要 work。任何一個壞，回 step 2 修 spec。
  5. 截圖加進 gallery
    用 playwright 或瀏覽器截全頁圖，存到 assets/shots/。在 architecture.html 的 #gallery 區段加新 card。
  6. Git commit + push
    這個情境正式成為 validated path，可以上 booth。同步更新 README + manifest（如果有用 booth log）。

06資料層限制 · 哪些故事還不能講

現有 RD skill 的覆蓋範圍是「讀寫設定 + 當前快照」。所有需要歷史聚合 / 即時 streaming 的故事都還做不出來，因為對應 API 還不存在。這個限制不是 dashboard-builder 的問題，是資料層尚未長出來。

🔴 完全擋住

趨勢 / 歷史聚合（"過去 7 天 throughput"、"本月流量 top 10"）

EnGenius Cloud 的 GET API 只回「現在」的快照，沒有「過去 N 天聚合」endpoint。所有「趨勢線」、「月度比較」、「年度報表」類 dashboard 都做不出來。

RD action: 補 history aggregation API（例如 /metrics/throughput?from=...&to=...&groupBy=day）
🔴 完全擋住

即時 streaming（"AP CPU 即時跳動"、"在線 client 即時列表"）

規格上 RD 有 subscribe_stat、subscribe_throughput、subscribe_client_list 等 25+ 個 op，但 3 個 troubleshoot skill 都還沒 scripts/。SKILL.md 列出來、references/ 規格寫好了，但無法執行。

RD action: 補 network-ap/gw/sw-troubleshoot skill 的 scripts/call_api.py（含 WebSocket / SSE 協定支援）
🔴 完全擋住

實體拓樸（"哪台 AP 接到 switch 哪個 port"、"線路斷在哪裡"）

需要 subscribe_fdb_list、subscribe_arp_list、subscribe_cable_diag — 也在 troubleshoot skill 內、也都還沒 scripts。所以我們的 topology_tree 只能畫邏輯階層（org → network → device），畫不出實體連線。

RD action: 同上
🟡 部分擋住

跨 org 成員聚合（"哪些人在多個 org 都有權限"）

get_org_memberships_overall 只能撈一個 org 的成員。要看「一個 user 在 5 個 org 各有什麼權限」需要拼接 5 次呼叫 + 用 email 比對。Dashboard 可以做（用 compute_fn 拼），但成本高。

RD action（nice-to-have）: 補 cross-org membership endpoint
🟡 部分擋住

裝置物理位置（"地圖上標出各分店設備"）

Network / device 沒有 GPS / address 欄位。hierarchy_view 名稱可能含地點線索（如 "Taipei"、"LA"），但需要 geocoding 推測。map widget 因此被延後，要等 RD 加 location 欄位。

RD action（nice-to-have）: device/network 加 location 欄位（address / lat / lon）

現在能做的 widget 類型

受限於上述限制，dashboard-builder library 目前聚焦 「當前狀態的多維呈現」：

  • ✅ 單一指標（kpi_grid）— "現在有 10 台設備"
  • ✅ 分類 / 構成（bar_list, donut）— "AP 9 / Switch 1"、"40% accessible / 40% partial"
  • ✅ 清單明細（table）— "列出所有 stale members"
  • ✅ 結構關係（topology_tree）— "Org → Network → Device 階層"
  • ✅ 單點時間軸（timeline）— "license 何時到期"（不是「過去到現在的趨勢」，是「現在到未來的點」）
  • ❌ 連續趨勢線（line_chart / sparkline）— 需要歷史 API
  • ❌ 即時跳動（live_meter / activity_feed）— 需要 streaming API
  • ❌ 地理（map）— 需要 location 欄位
