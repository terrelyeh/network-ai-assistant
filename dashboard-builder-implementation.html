# 主流程

    ↓
[1] 意圖解析 · LLM gate 1 (Haiku)
    ↓ {scope, time_range, metrics, audience}
[2] Widget 選擇 · LLM gate 2 (Sonnet)
    ↓ [{type, title, data_source}, ...]
[3] Query 生成 · 確定性 (LLM 不直接寫 query)
    ↓
[4] 資料取得 · 呼叫 primitive + Redis cache
    ↓
[5] Layout 排版 · grid layout engine
    ↓
[6] 渲染預覽
    ↓
[7] 使用者確認
    ↓
[8] 持久化儲存 (Postgres)
    ↓
[9] Refine loop ← 「再加一個 X」回到 [2]
⭐ 核心設計原則

① LLM 只在「判斷」用（解析、選擇）
② 不讓 LLM 寫 SQL（讓 backend 翻譯，安全 + 可 cache）
③ 限定 widget library 8 種（避免 hallucination）
④ 二段式 LLM（Haiku 解析 + Sonnet 選擇，平衡成本）

實作 Roadmap

📅 4 階段推進

PHASE 0 · 2 週

🔬 PoC（概念驗證）

目標：證明「自然語言 → dashboard」可行。
範圍：3 widget、mock data、單用戶、無持久化。
驗證：5 個 SMB IT 試用，看是否 get 到價值。

PHASE 1 · 6 週

🚀 MVP（最小可用版本）

目標：First real users。
範圍：8 widget 全做、接真實 telemetry、save/load、基本 refine。
驗證：3 個 design partner 每週實際使用。

PHASE 2 · 6-8 週

🏗 Production（GA-ready）

目標：對外正式上線。
範圍：multi-tenant、權限、performance、eval pipeline、mobile。
驗證：100+ users 穩定使用。

PHASE 3 · 12+ 週

⭐ Differentiate（拉開差距）

目標：競爭護城河。
範圍：cross-dashboard insights、export、custom widget plugin、API endpoint。
驗證：MRR 成長、客戶 NPS > 50。

主要風險

⚠ 風險與對策

風險機率影響對策
LLM 產 widget config 出錯高中嚴格 JSON schema validation + fallback widget
Widget 選錯類型中中Few-shot examples + eval 持續監控

LLM 成本失控中中Token budget / request、prompt cache、Haiku 為主

  • 合計：~8 人月

Operating · per 100 active users / 月

  • Hosting：~$80
  • DB + Redis：~$40
  • Monitoring：~$20

我的建議

✅ 推薦做法 vs ❌ 避免陷阱

✅ 推薦做法

  • 先 commit Phase 0 PoC（2 週、1-2 人）— 不要先想 production
  • PoC 過了才 hire 第二批人
  • Day 1 就建 eval pipeline（不要拖）
  • 限定 widget library 8 種
  • 二段式 LLM
  • 大部分 primitive、MCP、ring 系統可重用

❌ 避免陷阱

  • 讓 LLM 寫 PromQL → 會 hallucinate
  • 開放無限 widget 類型 → schema 會崩
  • PoC 就接 production data → 會被 telemetry 結構卡住
  • 沒做 eval 就改 prompt → 半年後 AI 退步
  • 用 Grafana plugin 走捷徑 → UX 不夠好

目標 & 成功標準

🎯 PoC 要證明什麼

⭐ 唯一目標

證明「自然語言 → dashboard」這個核心命題在 SMB IT 身上 work — 不是測試 production 級的可用性。

成功標準（必須全部達成）

  • 5 個 SMB IT 用 chat 描述需求，AI 都能產出可看的 dashboard
  • Widget 選擇正確率 ≥ 80%
  • 從輸入到 dashboard 出現 < 10 秒
  • 使用者 refine 一次成功率 ≥ 70%（「再加一個 X」）
  • 整個 PoC 跑下來成本 < $50 LLM API 費用

PoC 不做的事（明確 scope-out）

  • ❌ 多用戶 / 權限
  • ❌ 持久化儲存（每次 refresh 重來）
  • ❌ 真實 production telemetry（用 mock data）
  • ❌ Mobile 適配
  • ❌ 完整 8 widget（只做 3 種）
  • ❌ Eval pipeline（這留 Phase 1）

DAY 5

TAB 3 · EVAL PIPELINE

Dashboard 品質保證機制

AI 系統會悄悄退步，沒有 evals 就是賭博。 這頁定義「測什麼、怎麼測、CI 怎麼擋」— 大部分競品都沒做這件事。

為什麼必要

🚨 沒 eval 會發生什麼

📉 真實案例（業界常見）

第 1 個月：上線、AI 表現好、客戶滿意。
第 3 個月：Anthropic 推出新模型，產品換上 → AI 表現微弱下滑，沒人發現。
第 6 個月：工程師為了優化某 prompt 改了 2 行 → 該情境變好但其他 5 個情境變壞。
第 9 個月：客戶開始抱怨「AI 越來越笨」。
第 12 個月：團隊不敢動任何 prompt，半年內無功能更新。

Eval 解這個問題：自動化測試 AI 品質，每次改動都跑一次，退步立刻擋下。

測什麼

📐 Dashboard Builder 要評估的 4 個維度

D1 · 結構正確

Schema validation

確定性測試。LLM 輸出必須符合 widget JSON Schema：欄位齊全、type 對、enum 在範圍內。

方法：JSON Schema validator，過 = 通過。

D2 · 意圖匹配

Intent fields exact match

半確定性測試。golden case 標好預期 intent，比對是否一致（scope、time_range、metrics）。

方法：Field-by-field 比對 + 容錯（如時間範圍可接受 last_7d / last_week）。

D3 · Widget 選擇品質

LLM-as-judge

軟性評分。用另一個 LLM（Sonnet）打分數：「這組 widget 適不適合回答這個問題？」1-5 分。

方法：固定 judge prompt + 多次平均降噪。

D4 · 使用者意圖達成

End-to-end 完成率

真實使用者訊號。記錄每個 dashboard 生成後是「儲存」還是「丟棄」。儲存率高代表品質好。

方法：Telemetry 收集 + weekly 報告。

Golden Case 格式

📦 標準的測試案例長什麼樣

# tests/golden/case_001_weekly_health.json
{
  "case_id": "case_001_weekly_health",
  "description": "SMB IT asks for weekly health overview",
  "input": {
    "prompt": "我想看本週網路有沒有異常",
    "context": { "site_id": "demo", "user_role": "smb_it" }
  },

  # D1: schema validation - 自動跑

  # D2: 意圖匹配的 ground truth
  "expected_intent": {
    "scope": "single_site",
    "time_range": ["last_7d", "last_week"],  # OR list = 任一可接受
    "metrics_must_include": ["anomaly_count"],  # 必須有
    "metrics_may_include": ["client_count", "site_health_score"]  # 可選
  },

  # D3: widget 選擇預期
  "expected_widgets": {
    "min_count": 3,
    "max_count": 6,
    "must_include_types": ["line"],  # 至少有一個 trend chart
    "forbidden_types": ["heatmap"]  # 不該選太複雜的
  },

  # D3 補充：LLM-as-judge 的 quality threshold
  "judge_quality_min": 3.5,  # 平均分數至少 3.5/5

  # 元資料
  "annotated_by": "alice@network-ai-assistant.vercel.app",
  "annotated_at": "2026-05-15",
  "tags": ["smb_it", "weekly", "common"]
}

CI 整合

🔄 自動化執行流程

# .github/workflows/eval.yml
on:
  pull_request:
    paths:
      - 'prompts/**'
      - 'src/widget_selector/**'
      - 'src/intent_parser/**'
  schedule:
    - cron: '0 9 * * *'  # 每天早上 9 點全量跑

jobs:
  eval:
    steps:
      - name: Run eval suite
        run: |
          uv run python eval/run_evals.py \
            --cases tests/golden/*.json \
            --report-to=eval_report.json
      - name: Check thresholds
        run: |
          uv run python eval/check_thresholds.py \
            --report eval_report.json \
            --min-d1-pass 100% \
            --min-d2-pass 90% \
            --min-d3-quality 3.5 \
            --max-regression-vs-main 2%
🚦 Gate 規則

D1 (schema)：100% 必須過，沒得商量
D2 (intent)：≥ 90% golden case 過
D3 (judge)：平均品質 ≥ 3.5/5
Regression：對比 main branch，整體分數下降不可超過 2%

任一條件不過 → PR 被擋下，工程師必須修或解釋

TAB 4 · WIDGET LIBRARY

8 種 widget · 完整 schema

限定 widget set = 限定 LLM 選擇空間 = 提高準確度。 每個 widget 包含：視覺、JSON Schema、data source 約定、何時用。

為什麼限定 8 種

🎯 設計哲學

❌ 開放無限 widget 類型

  • LLM 選擇空間爆炸
  • 正確率快速下降
  • 每加一種要重訓所有 prompt
  • UX 不一致 / 維護地獄

✅ 限定 8 種 widget

  • LLM 選擇準確率 > 90%
  • UX 統一
  • Schema 全部 typed
  • 新增 widget 是「正式 release」事件，需審查
⭐ 跟 Anthropic Skills 哲學一致

Anthropic 的 Skills 系統用 progressive disclosure — 限定空間、依需求展開。Widget library 同理：少 = 強。

共用考量

🔧 所有 widget 都要處理的事

🔄 三種狀態

  • Loading：skeleton placeholder
  • Empty：「沒有符合的資料」+ icon
  • Error：顯示錯誤 + 「重試」按鈕

📱 響應式

  • min-width: 200px（小於就堆疊）
  • max-width: full grid column
  • Mobile: 單欄堆疊

♿ 可訪問性

  • 顏色不能單獨傳達訊息（要有 icon / text）
  • 所有 chart 要有 alt text 描述
  • 鍵盤可導航

🔄 即時更新

  • 每個 widget 設 refresh_interval（30s / 5m / 1h / never）
  • 背景刷新不打斷使用者
  • WebSocket 推送變更（可選）
