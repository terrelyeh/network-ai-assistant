解決什麼問題

非技術員工遇到網路問題（Wi-Fi 慢、印表機找不到、視訊卡）時，不用 call IT、不用查文件 — 直接問 AI chatbot。AI 用人話互動診斷，能自己解就解，解不了自動開單給 IT。

何時觸發

  • 員工在 chat 介面打：「Wi-Fi 很慢」「我連不上印表機」
  • 從公司 portal 點「網路有問題？」按鈕
  • 從 Slack / Teams bot 私訊問

AI 互動模式（與 Mode B/C 不同）

  • 教育式：解釋「訊號 −65 dBm 是普通強度」
  • 引導式：「請看您手機 Wi-Fi 圖示剩幾格」
  • 同理式：「對，我知道很煩。先試試…」
  • 有限度動作：只建議使用者自己做（換位置、重連）

解不了 → Auto Escalate

AI 把對話過程 + 已嘗試動作 + 自身位置 + 設備資訊整理成 ticket 開給 IT。IT 接到時不用再問背景，直接看就懂。

解決什麼問題

客戶或 monitoring 看到某台 AP / Switch 在 dashboard 顯示「offline」，需要快速判斷是真的離線還是誤報，並指出最可能的原因。

何時觸發

  • 客戶問「為什麼 AP-7F 顯示離線？」
  • Monitoring playbook 偵測到某設備 last_seen > 5 分鐘
  • Anomaly Panel 自動觸發（建議閾值：> 3 分鐘）

可能根因（依機率）

  • 網路不通（WAN 斷、上游 switch 掛掉）
  • 電源（PoE 端口失效、被拔）
  • 設備硬體故障（heartbeat 停但 cloud 還記得）
  • Cloud 端誤報（最後 telemetry 太久沒更新）
  • 網段隔離（VLAN 或 firewall 改了）

修復動作

⚠️ 大多需要現場處理（重插網線、檢查 PoE）。AI 只做診斷 + 產出指引，不做寫入。

範圍
1 client + 1 AP
Ring 等級
Ring 0 + Ring 1（可逆）
執行時間
~30 秒 + 10 分驗證
是否需現場
否（純遠端）

解決什麼問題

某個 wireless client 黏在訊號差的 AP 上不漫遊，導致使用者體驗差（語音卡、視訊延遲、傳輸慢），需要找出根因並遠端調整。

何時觸發

  • 使用者抱怨「我走到別的會議室還是慢」
  • 掃碼槍 / VoIP 電話在走動時斷線
  • Anomaly Panel 偵測到 client 長時間 RSSI < -75 dBm

可能根因（4 類）

  • RF 設計：cell 太大、TX power 過高、覆蓋重疊不足
  • Policy 缺失：min RSSI 沒設、band steering 沒開、802.11k/v/r 沒啟用
  • Client driver：client 端 OS/driver 不肯漫遊（網路端能誘導但不能強制）
  • 干擾：channel utilization 過高，roaming 沒效益

修復動作（依根因）

  • RF 設計 → 降 TX power 4-6 dBm Ring 1
  • Policy → 設 min RSSI = -72 + 啟用 802.11k Ring 1
  • Client driver → 標記為「網路端已優化」，需 client 升級
  • 干擾 → 調整 channel plan Ring 1

完整流程

解決什麼問題

SMB IT 不會設儀表板（不知該看什麼 metric / 不會 query 語法 / 不會排版）。AI 用講的就生出 4-6 widget 的儀表板，使用者預覽、確認、儲存。後續可 refine：「再加一個會議室趨勢」。

何時觸發

  • 「我想看本週網路健康」
  • 「show me top 10 sites by client churn」
  • 「哪些 AP 太忙？」「會議室訊號狀況」
  • 從現有 dashboard 點「+ 用 AI 加一個面板」

第三類 Skill（不是 playbook）

跟 Diagnostic / Monitoring playbook 結構不同：

  • 輸出不是 finding — 是 持久化的 UI artifact
  • 沒有 root-cause 分支 — 是 widget composition
  • 有 refine loop —「再加一個 X」
  • 可以儲存、分享、排程 email

Widget Library（限定 8 種）

  • 📊 KPI Card · 📈 Time Series · 📊 Bar Chart · 📋 Table
  • 🗺 Site Map · 🔥 Heatmap · ⚠ Alert List · 🟢 Status Grid

🔊 Sonos / 多播設備發現診斷 sonos_lan_discovery.yaml ★★★ 困難
範圍
controller + N hops + target
Ring 等級
0 + 1 + 可能 2（VLAN）
執行時間
~45 秒
是否需現場
有時要（跨廠商 hop）

解決什麼問題

App / 控制器找不到同 LAN 的多播設備（Sonos、AirPlay、Chromecast、Bonjour 印表機等）。涉及多 hop 拓撲，常常多個根因並存，是網管圈最頭痛的 case 類型之一。

何時觸發

  • 「我的 Sonos 在 App 裡看不到」
  • 「AirPlay 投影找不到電視」
  • 「Bonjour 印表機消失了」
  • 客戶報多媒體 / 智慧家居設備互通問題

可能根因（多重共存）

  • VLAN 分離：controller 與 target 不同 VLAN，發現流量過不去
  • STP 阻擋：路徑上某 port 在 blocking 狀態
  • Multicast 沒設好：IGMP snooping 沒開 querier、cross-VLAN mDNS gateway 沒設
  • IP 衝突：兩個設備搶同 IP
  • Port 錯誤：CRC 飆高、線路品質差
  • Coverage gap：問題在我們看不到的 hop（客戶用 Cisco core）

修復動作

  • VLAN → 把 port 移到正確 VLAN Ring 2
  • STP → 調整 port mode Ring 1
  • Multicast → 啟用 IGMP / mDNS gateway Ring 1
  • Coverage gap → 產出客戶 runbook，請客戶端工程師檢查

完整流程

9× Ring 0 1× Ring 1 1× Ring 2 2× LLM Gate ~$0.08/次
⭐ 為什麼這個 case 重要（架構壓力測試）

這個 case 同時把四個架構壓力施加到系統上：
① 跨設備診斷（multi-hop primitive、時間軸對齊）
② 多重根因（LLM 必須排序 + 承認 co-existing）
③ 看不到的 hop（capability probe + 客戶 runbook 機制）
④ Ring 2 高 blast radius 動作（強化 approval + cooling-off）
能完整跑這個 case 的架構，剩下的 playbook 都是它的子集。
