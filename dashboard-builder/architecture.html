你的專業 AI 網管

把 EnGenius Cloud 整套 API 變成 AI 可動手做事的能力 — 一個有 voice、有判斷、會記憶的 AI 同事，而不是又一個 chatbot、又一張 BI dashboard。

Claude Code 在對話中編排 目前 13 個 data skill（最小 primitive 單位、持續擴增）+ persona / design / playbook / memory / house-rules 5 層知識架構，依使用者問題即時組合資訊，自己判斷該回 文字 / dashboard / 動手操作。

所有 dashboard 都是 AI 即時組裝的 — 沒有預先寫好的模板，每次都是 AI 從 skills 即時拉資料、依情境組出當下最該呈現的視覺。

§01這是什麼 / 不是什麼

最容易誤解的點是「這套東西不在 EnGenius Cloud 網頁裡」— 它是在使用者自己的 LLM coding 環境（Claude Code）裡跑的 AI agent skill suite。

✅ 是 / ❌ 不是 ✅ 是：你在 Claude Code 終端機裡用自然語言對話，AI 透過 skill 呼叫 EnGenius Cloud API，回你文字 / 即時組裝 dashboard HTML / 動手操作雲端設定
❌ 不是：在 cloud.engenius.com 網頁裡開一個聊天視窗（那是另一條路線「Cloud-Embedded AI」，目前未推進）
❌ 不是：又一個 BI dashboard / SaaS 預設 view（dashboard 不是預寫的，是 AI 即時組）
❌ 不是：純「LLM + API」工具（我們有完整 persona / playbook / memory 讓 AI 像專業同事）

跟「LLM + API」純工具差在哪

其他工具給你的是 tool（會打 API，但每次都從零開始）。我們給你的是 colleague：

  • 有 voice — persona.md 規範語氣，不是冰冷的 datasheet
  • 有判斷 — 同一個問題會根據資料複雜度決定回 text 還是 dashboard 還是直接動手
  • 懂平台規則 — house-rules 紀錄 EnGenius 產品獨有的硬性規範與 quirks（規劃中）
  • 有思考骨架 — playbook 給 configure / troubleshoot / monitor 各自的 mental model（規劃中）
  • 會記得 — memory/<org-id>.md 跨 session 記錄客戶 quirks 跟 open follow-ups（規劃中）

§03每一層架構深度說明

上面架構圖的 5 個知識層 — 每個 .md 實際在寫什麼、AI 怎麼讀它、為什麼這個 layer 存在。Skill 那層已在 §02 + §05 詳述，這節聚焦在 AI 「同事化」的 5 個 layer。

§03.1 · persona.md · 怎麼跟使用者講話

它在寫什麼：358 行的 voice 守則 + escalation 矩陣 + 詞彙翻譯表。定義 AI 該怎麼回應 SMB 客戶 — 不是 RD 對 RD、不是 datasheet 朗讀，是「資深網管同事 explain 給老闆聽」的語氣。

三大 section（節錄概念）

  • Voice principles — 用「我幫你看了一下」開頭、不確定要說「不確定」、jargon 翻成 SMB 友善版（HVS → 健康分數 / RBAC → 權限角色）
  • Escalation matrix — 三種 output（text / dashboard / action）何時挑哪個的判斷規則
  • 詞彙翻譯表 — 60+ 條技術詞 ↔ SMB 友善說法對照

AI 怎麼用

所有 dashboard-builder skill 的 SKILL.md 標記 MANDATORY-loads persona.md — AI 在每次 invoke skill 前讀過、應用到當次對話。已在 2026-05-16/17 實測驗證（見 persona-test-results.md）。

§03.2 · design.md · 生成的 dashboard 長什麼樣

它在寫什麼：341 行視覺守則。三層自由度架構（locked tokens / flexible patterns / raw_html escape） + 9 條 hard prohibitions（遇到絕對停）。

9 條 hard prohibitions（摘要）

🚫 不准純黑 #000000（用 #1a2332）
🚫 數字不准用 font-mono（用 tabular-nums）
🚫 widget container 不准 box-shadow（chrome 留給 layout）
🚫 雙 theme 不准同頁混用
🚫 color hex 不准硬寫進 widget（一律走 token）
🚫 不准用 emoji 當主要視覺載體（語義不一致跨平台）
🚫 chart 不准 default seaborn / matplotlib 色票
🚫 KPI grid 不准超過一行 6 個欄位
🚫 raw_html 不准沒寫 raw_html_reason（自我擴張 library 必備）

AI 怎麼用

dashboard-builder skill 的 SKILL.md 也標 MANDATORY-loads design.md — AI 組 dashboard 前過一次守則。compose.py 在生 HTML 時驗證部分守則（例：token 使用）。違反 raw_html 規則時印警告，作為 library growth 信號。

§03.3 · house-rules.md · EnGenius 平台規則與規範 🔜 規劃中

將寫什麼：EnGenius 產品獨有的硬性規則與規範 — 都是平台層面的事實，不是主觀偏好。Persona 是 HOW（怎麼講話），house-rules 是 WHAT FACTS（這套系統的規矩是什麼）。例如：

  • HVS 分數定義 — 計算公式、區間意義（< 70 / 70–85 / > 85 各代表什麼）
  • License-Device 綁定規則 — AP-Pro license 必須綁 Pro 設備、Basic 不能跨 model 共用、co-term 規則、什麼能 stackable 什麼不能
  • Plan 限制 — BASIC vs PRO 哪些 endpoint 會 402 Payment Required、哪些功能要 PRO 才能用
  • 設備支援矩陣 — Dakota platform 不支援 spectrum scan、ESG 系列 vs Cloud 系列差異、哪些 model 支援 WiFi 7
  • 平台 quirks / 已知 edge case — Switch port PoE+ 並用衝突、某些 ACL 規則會被 Country 設定 override、設定組合不相容

AI 怎麼用

當使用者問「為什麼…」「我這個能做嗎」「會不會壞」時，AI 引用 house-rules 給「事實正確」的答案而不是「我猜」。例：

  • Q「為什麼 Gordon org 的 inventory 拿不到？」→ A「Gordon 是 BASIC plan，get_inventory 在 BASIC 會回 402」（依 house-rule: BASIC plan endpoint 限制）
  • Q「這張 AP-1YR-Basic license 可以給 EWS357AP 用嗎？」→ A「不行，EWS357AP 必須綁 AP-Pro license」（依 house-rule: license-device 綁定）

跟 persona 怎麼分

同樣是 AI 回答的「材料」，差別是：

  • Persona = HOW · 怎麼回答、用什麼語氣、什麼時候改 output 形式
  • House-rules = WHAT FACTS · 這套系統的硬性事實、規則、限制（不是主觀推薦）

為什麼還沒寫

需要 RD 整理（他們最清楚平台規則、license matrix、設備 quirks）+ 業務協助梳理（哪些限制客戶常踩、哪些 edge case 影響成交）。光看 code 拼湊容易漏，需要一份「official 化」的文件。Schedule 在 P2。

§03.4 · playbooks/ · 怎麼思考各種任務 🔜 下個 session

將寫什麼：3 個檔案，對應網管 3 大類任務 — 每個是 mental model 而非 step-by-step script。

檔案對應任務類內容類型
configure.md 設定類 (Manage) 「新分店要怎麼建」、「template 套用前該檢查什麼」的思考骨架
troubleshoot.md 診斷類 「連不上時要按哪個順序問問題」的「詢問三定問」框架
monitor.md 監控類 「健檢要看哪幾個維度」、「容量 plan 要看什麼 metric」的維度框架

關鍵設計：mental model 不是 script

絕對不寫 「first do X, then Y, finally Z」— 這會凍結 AI 判斷力 = 把 AI 變成 stored procedure = 殺掉 wedge。寫的是「思考骨架」「診斷維度」「該問的問題」— AI 自己依情境拼出步驟。

原則 · script vs mental model ❌ 不寫：「Step 1: 跑 get_inventory · Step 2: 跑 subscribe_ping · Step 3: ...」

✅ 該寫：「思考骨架 + 該問的問題 + 不要做的事」— 讓 AI 自己依情境拼步驟。

具體範例 · troubleshoot.md 將長什麼樣

# Troubleshoot Playbook · Mental Model

## 思考的 3 層
1. **Client side** — 是不是只有這台 / 這種裝置壞？
   - 典型：iOS 跟 Android 行為不同、舊裝置不支援 WPA3、距離太遠
2. **Radio side** — 是不是這台 AP / 這個 SSID 設定？
   - 典型：通道壅塞、power 太低、SSID isolation 設錯
3. **Uplink side** — 是不是上游 switch / gateway / WAN？
   - 典型：PoE 不足、uplink 線品質差、ISP 不穩

## 詢問三定問
1. 「多少裝置受影響？」（1 台 vs 一整個 SSID 差很多）
2. 「什麼時候開始的？」（持續 vs 突發）
3. 「有什麼東西**剛改過**？」（最常見的元凶）

## 不要做的事
- 不要一開始就 reboot AP（資料還沒蒐集就重開 = 把證據刷掉）
- 不要建議客戶換硬體（先 exhaust 設定面的可能性）

↑ 給 AI 思考的骨架，不限制具體該怎麼回。AI 還是依當下資料即時組合回應。

怎麼載入 · 條件式

跟 persona / design.md「每次 skill 觸發都 MANDATORY-load」不同，playbooks 是條件式載入 — AI 判斷使用者意圖屬於 configure / troubleshoot / monitor 哪一類，才 load 對應的 mental model。三個檔不同時 load 避免 context 浪費。

狀態

  • 🔜 下個 session 起跑（configure.md 跟 monitor.md 先做，troubleshoot.md 等 RD 補 dolphin URL 後再寫 — 因為 troubleshoot 強烈依賴 subscribe / rpc ops）

§03.5 · memory/<org-id>.md · 記得跟你講過什麼 📐 Spec ready · 🔜 post-launch impl

將寫什麼：跨 session 的記憶 — per-org-id namespace 的記事本。內容可能包含：

  • Open follow-ups — 「上次說要重啟 AP-7F 但客戶說沒空、約這週」
  • 客戶 quirks — 「這個 org 的 wifi-3 SSID 設定特別、不能套用標準 template」
  • 互動脈絡 — 「上次跟你討論過 license 續約策略，結論是分批續」
  • 已知問題 — 「2F switch 那個 port 5 一直 flapping，已通報但 RD 還沒修」

為什麼這個 layer 重要

沒 memory = AI 永遠是「一次性問答工具」。有 memory = AI 是「長期合作的同事」— 你不用每次重講上下文。這是「colleague 而不是 tool」定位的最後一塊拼圖。

資料夾結構 · 按 org_id namespace（規劃示意）

references/memory/
├── <org-id-1>.md       # Main_Org 的 memory
├── <org-id-2>.md       # Claudia 的 memory
└── <org-id-3>.md       # Senna 的 memory

每個 org-id 一份檔案（per-customer namespace）— SI / MSP 跨多個客戶時不會混淆。每次 session 開始按目前操作的 org_id load 對應檔案。

每份檔案內容 · 規劃中的範例

📐 Spec 鎖定（2026-05-18）· 實作 🔜 post-launch

Spec 已定下來，等真實 SI 用 1-2 個月、收集踩坑後再實作：

  • Namespace：per-org（不是 per-network / per-device — 太細）
  • Loading：Lazy — AI 第一次 resolve 到 org_id 時自動載入；不存在就跳過
  • Write 機制：Hybrid — SI 隨時手改 + AI session 結束時提議補充，絕不 silent write

  • Override 規則：per-customer memory 永遠 override universal 4 層

  • Sensitivity：v0 不存敏感事（投訴 / politics）— SI 自己用其他工具記

總結 · 為什麼這 5 個 layer 是「colleague 而不是 tool」的關鍵 Persona 給 AI voice、Design 給 AI 視覺品味、House-rules 給 AI 平台事實知識、Playbook 給 AI 思考骨架、Memory 給 AI 跨 session 記憶。每一層都對應一個「真實同事」的特質。少任何一層，AI 就退化成一個會打 API 的 chatbot。

§04AI 怎麼運作 · 即時組裝、不預寫

這是整套產品最重要的設計決策 — 所有 dashboard 都不是事先寫好的。AI 在收到使用者問題的當下，依情境從 skills 拉資料、自己判斷該回什麼形式、即時組裝出對應的視覺。

核心 punch line 沒有預先做好的 dashboard 模板。每次使用者問問題，AI 都是從目前可用的 skill 即時拼出當下最該呈現的資訊 — 同樣的問題在不同情境下會產出完全不同的 output。

三種 output 形式 · AI 自己挑

依 persona.md 的「升級條件矩陣」，AI 在每次回應前都會問自己：

Output 形式 適用情境 例子
📝 Text 資料簡單、結論明確、不需要視覺對比 「今天網路怎麼樣？」→「9 個 org 全綠燈、沒事，喝杯咖啡吧 ☕」
📊 Dashboard 資料多面向、需要對比、user 要 review 後決定 「我管的 5 個 org 健康狀態？」→ 即時組 KPI grid + alert timeline + offline 設備列表
🔧 Action 使用者明確要動手、含 confirmation gate 「都拿掉吧」→ delete_org_user_membership × N（每個 admin 角色獨立 gate）

同一個問題、不同情境 · 不同 output

S12「早安網路健檢」就是最好的示範 — 同樣是「今天網路怎麼樣？」，AI 依資料品質自己挑：

  • 全綠燈 → 30 字 text reply「9 個 org 全綠燈、沒事」
  • 1-2 個 minor issue → text + alert chip（不開 dashboard）「整體 OK，但 AP-LAB-03 從昨晚 offline」
  • 多重 issue → 即時組 dashboard（KPI grid + offline timeline + license expiry timeline）

Dashboard 是複雜度的 escape hatch，不是預設 output。

§05Skill = 最小 Primitive Unit

我們提供 primitives（最小可用單位），AI 自由組合。架構止於 skill 層 — 上面那些 layer（persona / design / playbook / memory / house-rules）由 AI 在 run-time 自己判斷怎麼使用。

目前 13 個 skill ready · 94 個 op（持續擴增）

Skills 是 Claude Code 的安裝單位。每個 skill 包含：

  • SKILL.md — op 定義（method / path / params / 描述）
  • references/<op>.md — 每個 op 的完整 schema + 範例
  • scripts/call_api.py — 5 行 delegation 到共用 framework

OpenAPI 公開 · AI-agent-friendly

我們把目前 13 個 skill 的 94 個 op 自動生成 OpenAPI 3.1 spec，這讓整套 skill suite 對 AI agent 特別友善：

  • AI 自動發現 — 任何 LLM agent 抓 openapi.json 就懂全部 API
  • Swagger UI 互動式 — api-docs.html 可點按、可 Try it out、可看 schema
  • Postman 一鍵 import — 整套 collection 自動建好
  • 各語言 SDK 自動產生 — 不用人工寫 wrapper
  • 跟 code 永遠同步 — script 從 SKILL.md 重生，文件不會飄掉

未來：不只 13 個 skill

13 是現在能用的 — 但 EnGenius Cloud 的 API 還有很多沒被包成 skill。我們已經盤點完所有 API 表面（96 個 op + 還有未開發的）：

  • 目前 47 個 op 完整可用（READ / MUT / DOWNLOAD）
  • 47 個 op pending RD 提供 URL pattern
  • 剩下還有未被組合過的 API → AI 可以基於現有 API 自己發想新 skill / 新 scenario

這是「食材 vs 菜色」的關係 — 我們提供 primitives（食材），AI 跟 PM 一起腦力激盪能煮出哪些菜色（scenario）。

§06Dashboard Output · 即時組裝 + 實例證明

當 AI 判斷該開 dashboard 時，會用 dashboard-builder skill 即時組裝。三種 output 形式（text / dashboard / action）之一 — 不是這套產品的主角，但視覺最直觀。

怎麼運作（簡介）

AI 根據對話結果寫出一份 spec JSON（描述要哪些 section、用哪些 widget、餵什麼資料），跑 compose.py 即時生成自包含的 HTML。每次都是即時組，不是從預寫模板挑一個出來。

📚 Appendix B · Glossary

這頁出現的術語對外讀者可能不熟悉，這裡一行解釋。

Falcon / Dolphin / LarkEnGenius Cloud 3 個後端服務代號 — Falcon = 管理系統 API（CRUD）· Dolphin = troubleshoot 服務（RPC / streaming）· Lark = membership invitation redirect
HVSHierarchy View Score — org 的整體健康分數，0-100
RPC / Subscribe / Polling三種 op 互動模式 — RPC 是一次性遠端命令（如 led_dance）· Subscribe 是串流訂閱（如 client_list 每 6 秒）· Polling 是輪詢 REST 模擬即時
Wedge product切入市場用的產品定位 — 楔子型產品，最小可贏的競爭差異點
Plugin / MarketplaceClaude Code 的安裝機制 — 用 /plugins 指令 install skill bundle
Persona / Playbook / Memory我們知識架構分層的詞 — persona 是 voice、playbook 是 mental model、memory 是跨 session 記憶
Spec JSONDashboard 的「配方」— 描述要哪些 section、用哪些 widget、餵什麼資料。AI 寫出 spec、compose.py 轉成 HTML
SMB IT / SI / MSPSMB IT = 中小企業的內部 IT · SI = system integrator · MSP = managed service provider（為多個客戶代管網路的廠商）
